# v0.1 launch plan

> End goal: a public URL anyone can visit that answers
> *"is it just you, or is it actually bad right now?"* — current, honest,
> auto-updating, presentable.

This document plans the remaining work toward that launch. Every stage
serves the end goal; anything that doesn't is out of scope.

## Locked design (do not re-debate)

- v0 = headline complaint rate + 90-day trend + top phrases + defection rhetoric
- Curated phrase list classifier (no ML)
- Same phrase list, both models
- Trend > absolute level
- Zero recurring cost (GH Actions free tier + JSON-in-repo + GH Pages)
- No accounts, tracking, or ads
- Dashboard: dark monospace, single-page, methodology co-equal with chart

These are settled. See `CLAUDE.md` and `docs/PIVOT-2026-05-06.md`.

## Stage 1 — Data quality consolidation

**Goal:** Corpus is rich and balanced; classifier is auditable;
release calendar is real.

**Status:** in progress (reddit re-run running, takes hours).

**Deliverables:**
1. Reddit historical re-run completes (background; no action needed)
2. Re-run `scripts/v0_classify.py` once reddit fills in
3. Verify release dates in `config/releases.json` against Anthropic + OpenAI
   announcement pages; flip `verified: true`. (User task, ~30 min.)
4. **Audit page** at `phase5/audit.html`: shows 50 random complaint records
   with the matched phrase highlighted. Lets you eyeball false-positive rate.
5. **Phrase list refinement** based on audit findings. One pass; don't iterate
   forever.

**Done gate:**
- `corpus_stats.py` passes (≥30% per family, ≥6 months)
- ≥3 verified release events in `releases.json`
- Audit page reviewed; obvious false positives addressed
- `data.json` regenerated with refined classifier

**Effort:** 1–2 sessions (depending on reddit re-run completion timing).

## Stage 2 — Dashboard depth + presentation

**Goal:** Dashboard reads as a complete artifact, not a sketch.

**Deliverables:**
1. **Per-source breakdown.** Show Claude/OpenAI panels with HN vs Reddit
   complaint rates side-by-side or as small subtotals. Tells the user whether
   the wave is broad or community-specific. Currently the rate is mixed across
   sources, which can mislead.
2. **Top permalinks per category.** For each top-N phrase, show 1–2 actual
   permalinks the user can click through to read. Confirms the headline isn't
   just regex bingo.
3. **Lemon palette swap.** When user provides the palette, replace 5 CSS
   custom properties in `:root` (`--lemon-rind`, `--lemon-flesh`, `--lemon-pith`,
   `--lemon-leaf`, `--lemon-zest`). Both `phase5/style.css` and `~/Desktop/lemon.html`
   need the swap; build_standalone.py rebuilds the desktop file.
4. **Mobile layout tightening.** Currently basic responsive — single-column
   below 1024px. Test at 375px (mobile), tighten typography and chart heights.
5. **Loading/empty states.** What happens if `data.json` is missing or sparse?
   Currently the dashboard shows "(no data)". Make it informative.

**Decisions to make in this stage:**

- **Per-source breakdown design.** Two options:
  - **Inline within each panel** — small subtotals beside the big number, e.g.
    "42% (HN: 31% · Reddit: 51%)". Compact, integrated.
  - **Stacked area on the trend chart** — color-code HN vs Reddit contribution.
    More visual, more chart real estate.

  Pros/cons + rec resolved when we get there. Default to inline first; promote
  to chart if texture warrants.

- **Permalink surface count.** 1, 3, or 5 per phrase?
  - 1 → minimal, lets you spot-check
  - 5 → richer but list gets long
  - Recommend 3 (oldest, most-recent, top-scored — spans the range).

**Done gate:**
- All five deliverables shipped
- Dashboard renders well at 375px / 768px / 1280px
- Visual review by user — passes the "looks intentional" test

**Effort:** 1 session, plus user time for palette decision.

## Stage 3 — Phase 4: GitHub Action automation

**Goal:** Data refreshes automatically; dashboard is always current.

**PRE-FLIGHT (required, see CLAUDE.md):**

Before writing the workflow, push a feature branch with a one-off workflow
that just `curl`s old.reddit.com from a GH Actions runner. Verify HTTP 200.
**If blocked, do not proceed with Phase 4 in its current form.** Pivot to:
- Run the cron from a different host (Cloudflare Workers? Self-hosted runner?)
- Solve PRAW account-side
- Use a residential proxy

**Deliverables (assuming pre-flight passes):**
1. **Workflow YAML** at `.github/workflows/refresh.yml`:
   - Cron: every 4 hours (configurable)
   - Steps: scrape (HN + arctic_shift + reddit unauth) → classify → commit data.json
   - Idempotent: re-running same window produces same output
2. **Storage strategy:** `data.json` lives on a `data` branch (clean main history).
   Pages serves from a build that pulls from both branches.
3. **Rate-limit safety:** sleep budget across all scrapers ≤ 100s per run; respect
   `x-ratelimit-remaining` headers; backoff on 429/403.
4. **Failure visibility:** action emails/notifies on failure (built-in GH UI;
   no extra config needed).
5. **Reclassify trigger:** when `config/complaint_phrases.json` changes,
   the next cron re-runs the classifier on the existing corpus from scratch
   so old weeks reflect the new list.

**Decisions to make:**

- **Cron cadence.** 4 hours is a starting guess.
  - 1 hour: more current; more cron commits in git log; more API hits.
  - 4 hours: balance.
  - 24 hours: lowest cost; freshness lags noticeably.
  - Recommend 4 hours for v0.1; adjust based on observed traffic.

- **Storage layout.**
  - `data` branch (clean main): more setup; clean separation.
  - `phase5/data.json` on main: simpler; commit log gets noisy.
  - Recommend `data` branch.

**Done gate:**
- Workflow runs successfully twice on schedule
- `data.json` is committed and dashboard reflects new data
- Failure notifications tested (kill a scraper deliberately, verify alert)

**Effort:** 1–2 sessions.

## Stage 4 — Public deployment

**Goal:** Dashboard has a public URL; methodology is publish-ready.

**Deliverables:**
1. **GitHub Pages config.** Project page at `mobetter20.github.io/lemon/`,
   serving from `phase5/`. (If we use a `data` branch for data, the Pages
   build needs to include both — small workflow tweak.)
2. **Custom domain (optional).** Defer for now; default subdomain is fine.
3. **SEO/social meta tags.** `og:title`, `og:description`, `og:image`,
   `twitter:card`. Honest about what the dashboard is.
4. **Open-source the repo.** Flip `mobetter20/lemon` from private to public.
   Methodology and phrase list become auditable.
5. **README polish for public-readable.** Currently the README is OK but
   leans agent-instructional. Tighten for first-time visitors.
6. **`humans.txt` or `/about`.** Brief project authorship + intent. Optional.

**Done gate:**
- Public URL loads
- Mobile renders correctly on real device
- Methodology is clear without prior context
- Repo is public; CLAUDE.md is acceptable to be world-readable
  (it currently is; sanity-check)

**Effort:** 1 session.

## Stage 5 — Soft launch + calibration

**Goal:** Prove the pipeline runs reliably for ~30 days before any public
announcement.

**Deliverables:**
1. **30-day calibration window.** Cron runs daily/4-hourly without intervention.
   Watch trend lines for continuity, errors for silent failures.
2. **Monitoring sanity-checks.** Once a week: glance at the action logs,
   check `data.json` for sane numbers, eyeball trend.
3. **One iteration of phrase list refinement** if obvious gaps appear in
   live data.
4. **No public announcement until day 30.** The brief was explicit about
   this. Per `docs/PIVOT-2026-05-06.md`, the project's own launch shouldn't
   be theatrical — it mocks marketing theater.

**Done gate:**
- 30 days of cron data with no missing days
- Trend lines show sensible variation, not noise spikes
- User looks at it and says "yes, this answers the question"

**Effort:** mostly unattended; ~30 min/week of manual checking.

## Stage 6 — Public launch

**Goal:** Project goes live.

**Deliverables:**
1. **Announce.** Where: HN ("Show HN: lemon — is it just you?"), Bluesky,
   personal site if you have one.
2. **Honesty in framing.** The dashboard's methodology section should
   already disclose everything; the announcement should match its tone
   (no overclaiming, no "AI is dying" framing — just the question).
3. **Watch the response window.** First 48 hours: check feedback for
   real critique vs noise; address legitimate methodology issues.

**Effort:** 1–2 hours announce + ongoing monitoring.

## What's explicitly out of scope for v0.1

These are good ideas; defer to v0.2 or later.

- Per-subreddit drill (r/ClaudeAI vs r/LocalLLaMA voice differ)
- Anomaly detection ("today's rate is 2σ above 30-day baseline")
- API for programmatic access to data.json
- Multi-language UI
- Email digest / RSS feed
- The v1 taxonomy work (archived; explicit reopening required)
- Twitter/X coverage (still gated by paid API)
- Bluesky scraping (could be added in Phase 4 cron if cheap)

## Critical path (sequential)

```
Reddit re-run finishes → Stage 1 (data) → Stage 3 (Phase 4 cron + pre-flight)
                          ↓
                       Stage 2 (UX) → Stage 4 (deploy) → Stage 5 (calibrate) → Stage 6 (launch)
```

Stage 1 + Stage 2 can run in parallel. Stage 3 needs Stage 1 done (clean data
to test the cron). Stage 4 needs Stage 3 done. Stage 5 starts the day Stage 4
ships.

## Estimate

- Stage 1: 1–2 sessions (mostly waiting on reddit re-run)
- Stage 2: 1 session
- Stage 3: 1–2 sessions (with PRAW pre-flight; one extra session if pre-flight fails)
- Stage 4: 1 session
- Stage 5: 30 days, mostly passive
- Stage 6: 1–2 hours

**Total active time to public launch: ~5–7 sessions plus 30-day calibration.**

## When this plan should be revisited

- If the GH Actions IP probe fails (Stage 3 pre-flight) → Stage 3 needs replanning
- If Stage 5 calibration reveals systematic issues with the phrase list → loop
  back to Stage 1
- If user changes scope (e.g., wants to bring v1 taxonomy back) → re-baseline
  this plan
